零、这是什么？为什么它存在？

一句话：NanoResearch 是一个端到端自动化科研智能体。你给它一个研究主题（比如"用知识蒸馏加速BERT推理"），它自动完成文献调研 → 假说生成 → 实验设计 → 代码生成 → GPU 运行 → 结果分析 → 图表绘制 → LaTeX论文撰写 → 自动审稿修订，最终输出一篇可提交的论文 PDF。

它和 ChatGPT"帮我写论文"有什么区别？

核心区别：它真的做实验。

ChatGPT 写论文时，数字都是编的。NanoResearch 会：

  1. 自动生成完整的 PyTorch 训练代码（model.py, dataset.py, trainer.py, main.py...）
  2. 在你的 GPU 上或 SLURM 集群上实际运行训练
  3. 收集真实的 metrics.json（准确率、损失、消融结果...）
  4. 用真实数据生成图表和论文表格
  5. 论文中每一个数字都能追溯到实验输出文件

术语表（新手必读）

术语含义
Agent一个自主执行特定任务的智能体程序。本项目中有 9 个 Agent，各负责一个阶段。
Pipeline流水线。多个 Agent 按固定顺序串联，前一个的输出是后一个的输入。
Orchestrator编排器。控制 Pipeline 的调度、重试、恢复，是整个系统的"大脑"。
LLM大语言模型（如 GPT、Claude、DeepSeek、Gemini）。Agent 通过 LLM 完成推理任务。
ModelDispatcherLLM 调度器。不同阶段可以配置不同的 LLM 模型，Dispatcher 负责路由和调用。
ReAct一种让 LLM 交替进行"推理"和"行动"（调用工具）的模式。
MCPModel Context Protocol。一个标准化的工具调用协议，让 Agent 可以搜索 arXiv、查 Semantic Scholar 等。
Blueprint实验蓝图。PLANNING 阶段生成的实验方案（数据集、基线、指标、消融设计）。
Workspace工作空间。每次研究会话的隔离目录，存放所有文件和状态。
Manifest清单文件 (manifest.json)。记录每个阶段的状态（pending/running/completed/failed）。
Quick-Eval快速评估。缩小模型+少量 epoch+少量数据运行，快速获取近似指标。
Dry-Run干运行。只初始化模型、加载数据、前向传播一个 batch，验证代码无错误。
PydanticPython 的数据验证库。所有阶段间传递的数据都有 Pydantic Schema，确保格式正确。
Mixin一种 Python 多继承模式。把大类的功能拆成多个小类（Mixin），然后组合在一起。
SLURM高性能计算集群的作业调度系统。通过 sbatch 提交 GPU 训练任务。
Grounding接地/证据绑定。确保论文中的数字来自真实实验结果，而非 LLM 编造。

两种 Pipeline 模式

Standard Pipeline（6 阶段）— 适合快速原型

IDEATION → PLANNING → EXPERIMENT → FIGURE_GEN → WRITING → REVIEW

ExperimentAgent 一人负责：代码生成 + 环境搭建 + 执行 + 分析。

Deep Pipeline（9 阶段）— 适合严肃研究

IDEATION → PLANNING → SETUP → CODING → EXECUTION → ANALYSIS → FIGURE_GEN → WRITING → REVIEW

4 个专业子 Agent 分别处理环境搭建、编码、执行、分析。更精细的控制。

三、Pipeline 编排层详解

类比：编排器就像工厂的生产线管理员。它不做具体的事，但它决定"下一步该谁干"、"出了问题怎么重试"、"停电恢复后从哪继续"。

3.1 类继承结构

BaseOrchestrator(ABC)                     # 抽象基类: 所有共享逻辑
├── PipelineOrchestrator                  # Standard 6 阶段实现
├── DeepPipelineOrchestrator              # Deep 9 阶段实现
│   └── UnifiedPipelineOrchestrator       # 推荐入口（等于 Deep）

3.2 BaseOrchestrator 核心机制

a) 子类必须实现的 3 个方法

方法作用返回
_build_agents()创建每个阶段的 Agent 实例dict[PipelineStage, Agent]
_get_processing_stages()返回有序阶段列表list[PipelineStage]
_prepare_inputs(stage, topic, accumulated, last_error)为每个阶段组装输入参数dict

b) run(topic) 主循环 — 系统的心脏

  1. 崩溃恢复：检查 manifest 中 status="running" 的阶段 → 重置回 "pending"（防止上次崩溃卡死）
  2. 逐阶段执行：
    • 已完成的阶段 → 跳过，直接加载之前保存的输出
    • config.skip_stages 中的阶段 → 跳过
    • 其他 → 调用 _run_stage_with_retry()
  3. 跨阶段验证：
    • PLANNING 后：检查 blueprint.hypothesis_ref 是否存在于 ideation 的假说列表
    • EXPERIMENT 后：检查实验指标是否在 blueprint 中定义过
  4. Blueprint 语义验证：PLANNING 后执行 6 项检查（指标非空、有 primary、方向一致性...）
  5. 成本汇总：保存 logs/cost_summary.json
  6. 后处理：Deep Pipeline 自动 export 到清洁输出目录

c) 重试机制

指数退避：delay = 5s × 2attempt，上限封顶。默认最多重试 2 次。

过程：标记阶段 running → 调用 agent.run() → 成功则标记 completed 并保存输出 → 失败则记录错误、等待后重试 → 最终失败标记 FAILED

四、状态管理与工作空间

4.1 状态机 (PipelineStateMachine)

类比：状态机就像一个"只能往前走"的棋盘。IDEATION 只能走到 PLANNING，PLANNING 只能走到 EXPERIMENT/SETUP...任何阶段都可以走到 FAILED。force_set() 是"作弊模式"，用于恢复时跳到任意位置。
  • transition(target)：正常前进，非法转换抛异常
  • force_set(stage)：跳过验证直接设置（记录审计日志），用于检查点恢复
  • fail()：从任何非终态转入 FAILED
  • 终态：DONE, FAILED

4.3 原子写入保障

所有关键文件（manifest.json, progress.json, metrics.json）使用临时文件 + os.replace()原子写入。意思是：要么写入完整成功，要么完全不写。即使中途断电也不会产生"写了一半"的损坏文件。

5.3 重试与容错

  • 瞬态错误自动重试：timeout, 429, 502, 503, 504, connection error → 指数退避最多 5 次
  • JSON 模式回退：模型不支持 json_object → 退化为 prompt-only JSON
  • Thinking 模型适配：o1/o3 等推理模型 → system prompt 合入 user message；自动清理 <think> 块
  • max_tokens 兼容：先尝试 max_completion_tokens，不支持时回退 max_tokens

六、Stage 1: IDEATION 文献调研

Agent：IdeationAgent = 2 个 Mixin (_IdeationSearchMixin 负责搜索, _IdeationHypothesisMixin 负责分析) + BaseResearchAgent

做什么：从研究主题出发，搜索文献 → 发现空白 → 生成假说 → 提取证据

搜索策略（多源并行+迭代补充）

  1. 生成搜索词：LLM 生成 5 个多样化查询（同义词、不同粒度）
  2. 4 源搜索：OpenAlex（全文索引, 5 req/s）→ arXiv（9 个分类, 0.3 req/s）→ DuckDuckGo → PapersWithCode
  3. 去重：paper_id 或 title 小写匹配
  4. 过滤排序：过滤会议条目 → 相关性评分(>0.30) → 混合排序（高引+近期）
  5. 引用扩展：从 Top-K 论文的参考文献中雪球采样（citation_count ≥ 20）
  6. 全文增强：下载 Top-4 论文 PDF → 提取 method_text + experiment_text
  7. 覆盖度评估：LLM 评分 1-10 → 建议缺失方向 → 补充搜索（最多 2 轮）

分析与假说生成（ReAct 模式）

LLM 搭配搜索工具，最多 2 轮循环：

  • 总结领域现状 + 主流方法
  • 识别 3-5 个研究空白（具体的，不是空泛的"需要更多研究"）
  • 提出 2-4 个可验证假说 + 新颖性论证 + 可行性评估
  • 选出 1 个最佳假说 + 理由

证据提取（严格无编造）

从论文摘要中精确提取定量指标：ExtractedMetric{paper_id, dataset, metric_name, value, method_name, higher_is_better}

铁律：只提取摘要中明确出现的数字。不估算，不推断，不编造。摘要没有数字的论文直接跳过。

七、Stage 2: PLANNING 实验规划

Agent：PlanningAgent

做什么：把选中的假说转化为完整的、可执行的实验方案

八、Stage 3: EXPERIMENT 实验执行

两种模式：Pipeline 模式（默认，结构化5轮迭代）和 ReAct 模式（LLM 自由驱动）

8.1 Pipeline 模式详解

Round 1（基线轮）

  1. 项目规划：LLM 生成文件列表 + 接口契约 + 依赖关系
  2. 代码生成：LLM 并行生成每个文件（main.py 必须支持 --dry-run 和 --quick-eval）
  3. 质量保障：Import 一致性检查 → Black 格式化 → Smoke Test（导入验证）
  4. Dry-Run：创建 venv → 安装依赖 → 执行（5 轮修复循环，每轮：执行→失败→解析 Traceback→LLM 生成补丁→应用→重试）
  5. Quick-Eval：缩小模型 + 3-5 epochs → 采集 metrics.json（同样 5 轮修复循环）

Round 2-5（迭代轮）

  1. LLM 生成改进假说（必须：只引用实际存在的文件、不重复之前的尝试、无新想法则终止）
  2. Search-Replace 编辑引擎（4 层模糊匹配）：
    1. 精确子串匹配
    2. 行尾空格忽略匹配 (rstrip)
    3. 锚定跨度匹配（首行+末行锚定，中间允许 8 行差异）
    4. 定义块匹配（识别 def/class 整块替换）
    回退到全文件重写（含截断检测：新内容 < 30% 原长则拒绝）
  3. 重新执行 Dry-Run + Quick-Eval
  4. FeedbackAnalyzer 分析本轮结果（归因分类 + 终止条件检查）

FeedbackAnalyzer（反馈分析器）

检查说明
指标摘要提取 main_results 中的数值指标
改进 Delta与上一轮对比
训练动态收敛速度(fast/normal/slow)、过拟合检测(train_loss < 0.5*val_loss)、损失稳定性
LLM 归因分类为 data_issue / training_strategy / architecture / hyperparameter / implementation_bug
终止条件目标达成 / 最大轮次 / 高原(patience) / 假说重复 / 退化(低于最佳 5%)

十三、WRITING 论文撰写

Agent：WritingAgent = 6 个 Mixin

做什么：从实验结果 + 文献 + 图表 → 完整 LaTeX 论文

核心创新: Per-Section Context (每节独立上下文)

问题：传统方法给每个 section 发送全量上下文 (~30-40K tokens)，浪费且容易幻觉。
解决：每个 section 只获取相关信息 (~8-15K tokens)。Introduction 需要 survey+gaps，Method 需要技术细节，Experiments 需要数据集+指标。互不干扰。

核心创新: Contribution Contract (贡献合约)

  1. Introduction 生成后 → 提取 \begin{itemize}...\item... 中的贡献声明
  2. 分类每条声明：method / component / empirical
  3. 注入后续 section：Method 必须有技术细节，Experiments 必须有证据，Conclusion 必须有总结

Grounding（证据绑定）

GroundingPacket 包含规范化的实验结果，用于：

  • Main Results 表格：baselines 在前 + proposed method 在后 + 自动加粗最佳值 + ±std
  • Ablation 表格：variants + full model + 只显示有数据的列
  • 无数据时：生成 scaffold 表（所有单元格 "--"），不编造

写作 Prompt 工程（关键规则）

Section关键规则
Abstract5 部分结构, 150-250 词, 1 段, 无引用, 无图表引用
IntroductionKaiming He 三段式(问题→方法→贡献), "N components"必须列出恰好 N 个
Method每组件用 \subsection{}, 每个操作有编号方程, 含复杂度分析
Experiments用 \paragraph{Finding Title} 组织(不写报告式), 用 "--" 标不可用数据
Conclusion引用具体数字, 具体局限(非泛泛而谈), 2-3 个可行未来方向

反 AI 写作检测

禁用词：delve into, leverage, utilize, harness the power of, pave the way, plays a crucial role...

禁用模式：formulaic rule-of-three, binary contrasts, excessive em-dashes, hedging pileup

检测：check_ai_writing_patterns() 扫描 16 个禁用短语 + 过度使用的 AI 词汇 + 重复段落开头词

十四、REVIEW 审稿修订

Agent：ReviewAgent = 7 个 Mixin

做什么：自动审稿 → 一致性检查 → 迭代修订（收敛控制）

评分标准

9-10出版就绪7-8扎实，小问题
5-6显著问题但可修3-4需大幅重写
1-2根本性缺陷分数反映严重性, 非数量

5 项自动一致性检查（无 LLM, 确定性）

  1. \ref{X} 没有对应的 \label{X}
  2. 重复的 \label
  3. 摘要声明 vs 正文数字不一致
  4. 贡献条目数量与正文描述不符
  5. 图/表 label 未被引用

修订循环算法（核心）

初始化:
  prev_avg_score = 各节分数平均值
  stall_count = 0     # 连续无改善轮数
  bp_revert_count = 0 # 回滚次数

循环 (最多 5 轮):
  1. 找出所有低分节 (score < 8)
  2. 对每个低分节: LLM 生成修订版本
  3. 应用修订 → 检查 LaTeX 结构是否被破坏
     - 如果破坏: 尝试自动修复 → 仍然破坏则 回滚 (bp_revert_count++)
     - bp_revert_count >= 2 → 强制停止
  4. 重新审稿评分
  5. 分数退化处理:
     - 对退化的 section → Meta-Refine:
       a) LLM 诊断失败原因
       b) 生成额外约束
       c) 重试修订 (带约束)
       d) 重新评分: 改善则接受, 否则回滚该 section
  6. 收敛检测:
     new_avg_score = 新平均分
     improvement = new_avg_score - prev_avg_score
     如果 improvement < 0.3:
       stall_count++
       如果 stall_count >= 2: 强制停止!
     否则:
       stall_count = 0
     prev_avg_score = new_avg_score

图表保护（两层防御）

  1. Prompt 明确告诉 LLM："PRESERVE all \begin{figure}...\end{figure} blocks"
  2. 代码检测修订后丢失的图表 → 自动重新附加到 section 末尾

Grounding 注入

修订时告诉 LLM 哪些数字是真实的、不可修改、不可四舍五入、不可编造。

面试可能追问 Q&A

Q: 为什么不直接用 LLM 修？

A: 三个原因：(1) 成本——每次 LLM 调用消耗几千 token，修一个 FileNotFoundError 可能花 $0.05，确定性方法免费；(2) 速度——确定性修复毫秒级，LLM 修复需要 3-10 秒网络延迟；(3) 可靠性——路径替换这种确定性操作 100% 正确，LLM 有幻觉风险。

Q: 如果 LLM 修复引入了新 bug 怎么办？

A: 每次修复后立即 py_compile.compile() 验证语法。语法不通过直接回滚到修复前的快照。修复历史记录在 repair_journal.json 里，如果同一个错误签名连续出现 2 次（repeat_count）就标记为 NON-RETRYABLE，避免死循环。

Q: 最多修几轮？

A: dry-run 和 quick-eval 各最多 5 轮修复循环。5 轮内没修好就放弃该轮迭代，但不终止整个 pipeline——FeedbackAnalyzer 会分析失败原因，下一轮迭代可能换个假说重新来。

Q: 怎么防止 pip install 安装恶意包？

A: 三层限制：(1) 模块名正则验证 [A-Za-z0-9_.-]+；(2) 如果 code_dir 下有同名本地模块/包，跳过 install；(3) ExperimentExecutionPolicy 有包数量上限（默认50）和可选白名单。

面试可能追问 Q&A

Q: 为什么用 Welch t-test 而不是普通 t-test？

A: 普通 t-test 假设两组方差相等（方差齐性），但 ML 实验中不同方法的方差通常不同（比如我们的方法方差大，baseline 方差小）。Welch t-test 放松了这个假设，用 Welch-Satterthwaite 公式计算修正的自由度。

Q: df ≤ 30 时为什么不用精确计算？

A: 精确计算需要 t 分布的 CDF，SciPy 的 scipy.stats.t.sf() 内部用了 Incomplete Beta Function。不引入 SciPy 的话只能用查表法。我用 df=10 的临界值——因为 df 越小分布越"胖"、p 值越大。实际 df ≥ 10 时用 df=10 的临界值会高估 p 值（即"更保守"），不会出现假阳性。

Q: 退化运行检测为什么要 ≥3 epoch？

A: 1-2 个 epoch 可能是初始化阶段，全零是正常的（比如 warmup 阶段 loss 还没 backprop）。3 个 epoch 以上全零基本可以确认是"静默失败"——比如 batch 内部 key 不匹配导致梯度为零但不报错。

Q: 过拟合检测用的什么算法？

A: 项目里有两套过拟合检测，用在不同地方：

  • training_dynamics.py（我写的分析模块）：取训练最后 1/3 的 val_loss 和 train_loss，分别做最小二乘线性回归算斜率。如果 val_loss 斜率 > 0.001（上升）且 train_loss 斜率 < -0.001（下降），就判定过拟合。阈值 0.001 是为了过滤噪声波动。
  • feedback_analyzer.py（同事写的迭代反馈）：更简单的启发式——如果 final_train_loss < 0.5 * final_val_loss，即训练损失不到验证损失的一半，就判定过拟合。

线性回归实现：slope = Σ(xi - x̄)(yi - ȳ) / Σ(xi - x̄)²，纯 Python 无 numpy。

# _kill_process_tree(pid: int) → None

# 问题: Python venv 在 Windows 上产生双进程:
#   venv/Scripts/python.exe (包装器)
#     → anaconda3/python.exe (真正的解释器)
# 只 kill 外层 → 内层变孤儿 → 继续占 GPU

# 解决方案:
if platform == "Windows":
    # taskkill /T = 终止进程树(所有子进程)
    # taskkill /F = 强制终止
    subprocess.run(["taskkill", "/T", "/F", "/PID", str(pid)])

else:  # Unix/Linux
    # 递归杀死进程树
    children = pgrep -P {pid}  # 获取直接子进程
    for child_pid in children:
        _kill_process_tree(child_pid)  # 递归
    os.kill(pid, signal.SIGKILL)       # 最后杀父进程

# 为什么不用 os.killpg()?
# → killpg 杀的是进程组(process group)
# → 如果子进程继承了父进程的 pgid
# → killpg 会连 Pipeline 主进程一起杀掉!

Q: 为什么用 Popen 而不是 subprocess.run？

A: subprocess.run() 不暴露 PID，超时时只能 proc.kill() 杀单个进程。Popen 可以拿到 proc.pid，配合我的 _kill_process_tree() 杀整棵进程树。这是从 subprocess.run 改为 Popen 的根本原因 (BUG-227)。

Q: 为什么不用 psutil 库？

A: psutil 是第三方依赖，我们希望部署尽量轻量。taskkill 和 pgrep 是操作系统自带命令，不需要额外安装。而且实现只有 ~40 行代码，没必要为此引入一个大库。

Q: 你在这个项目中遇到的最大技术挑战是什么？

A: "最大挑战是修复策略的优先级设计。一开始我们把所有错误都丢给 LLM 修，但发现两个问题：(1) 成本高——一轮修复要消耗上万 token，5 轮下来光修复就花好几毛钱；(2) LLM 修复不稳定——有时候修了这个 bug 引入另一个。后来我梳理了测试过程中积累的失败 stderr，发现大部分错误是确定性可修的（路径不存在、依赖缺失、参数不对），真正需要理解代码语义的是少数。所以我设计了 6 层优先级链：先试确定性方法，全部失败再用 LLM。这样既省了大量 API 调用成本，修复速度也从秒级降到了毫秒级。"

Q: 你对这个系统的架构有什么改进想法？

A: "两个方向：(1) 修复策略可以更智能——目前是固定优先级链，但实际上不同类型的错误有不同的最优修复路径。可以用一个轻量分类器先判断错误类型（路径/依赖/语法/逻辑），然后直接跳到最可能成功的层级，跳过不相关的层。(2) 结果分析可以更深入——目前的统计检验只做了 t-test，但 ML 实验经常有非正态分布（比如 F1 分数在高分端有天花板效应），可以加入 Wilcoxon signed-rank test 做非参数检验。"

Q: 为什么统计模块不直接用 SciPy？

A: "三个原因：(1) 部署简化——SciPy 编译依赖比较重（需要 BLAS/LAPACK），在某些集群环境下安装困难；(2) 透明性——手写实现我能精确控制每一步的行为，比如 p 值近似我故意选了保守方案（df=10 查表法高估 p 值），这比黑盒调用更可控；(3) 实际需求有限——我们只需要 t-test、效应量、Bootstrap CI 三个功能，写起来也就 150 行代码，不值得引入整个 SciPy。"

Q: 怎么保证修复不会越修越差？

A: "三层保障：(1) 快照回滚——每次修复前记录文件的 SHA256 和原始内容，语法检查失败立即回滚；(2) 重复计数——如果同一个错误签名（stderr 最后一行的前 240 字符）连续出现 2 次，标记 NON-RETRYABLE 跳过；(3) 修复日志——所有修复操作记录在 execution_remediation_ledger.json，包含 kind/status/scope/files/details 字段，方便事后审计和复盘。"
